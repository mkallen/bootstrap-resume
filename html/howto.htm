JSP 2.0 Templating

March 21, 2012

Ever run into an issue with copy and pasting the same code around on multiple web pages? This tutorial I put together will show you how to utilize JSP 2.0 tags to template out your busy UI!

Creating a UI Framework using JSP 2.0 Tag Libraries

Terms to know:

1.       Page Template – a reusable page layout defined in a JSP 2.0 .tag file

2.       UI Component Template – a reusable tag that defines a ui component such as a label, table, or textbox that hides complexity and simplifies use.

Why create a framework of page templates?

Say you have multiple different pages in different .jsp files that look very similar:

   

If we wanted to change the styling and layout of the three pages as they are now, we would have to separately edit all three files. There is a way to use JSP 2.0 to only edit one file and change all three.

To use our template we simply import our tag library (denoted in purple), define our page to use the specific template we want (denoted in light blue), and put our content within the jsp:attribute tags that our template expects. Notice that with page content being defined within the attribute tags, the developer of the page simply using the template does not have to be concerned with where that content will be placed on the page. The template takes care of that. If a hundred pages use a single template and we need to swap where content goes or what .css file the page should use only the single template would need to be edited.

Templates can be as simple or as complicated as needed and as many or as few custom attributes can be defined as you want.

Why create a framework of ui components?

Say, we have a ui component tag. It can be a <label>, a <select>, or even a <div>. But what if we want all instances of every <label> in our system to look the same? What if we wanted every <select> to be a dojo select that uses specific css classes every time? Wouldn’t it be great if we could distill a complex dojo table into a single custom JSP 2.0 .tag file that abstracted out all that javascript code so that a developer is only concerned with what data they are giving the table and what columns they want to show?

Imagine a tag like so:

<ui:example-table-json tableHeader="Supply Class" gridDataIdentifier="supplyClass"

            jsonTarget="../tag-test/supplyClasses" gridLayoutColumns="${gridLayout}"

            height="300px" width="500px" id="jsonTable" storeId="jsonStore"></ui:example-table-json>

That would give us this:

Let’s define example-table-json.tag:

<%@ attribute name="id" required="true" description="id for the grid"%>

<%@ attribute name="storeId" required="true" description="id for the store"%>

<%@ attribute name="height" required="true"

       description="height for the grid"%>

<%@ attribute name="width" required="true"

       description="width for the grid"%>

<%@ attribute name="tableHeader" required="false"

    description="header label to appear above the grid"%>

 

<%@ attribute name="gridLayoutColumns" rtexprvalue="true" required="true"

       description="array of GridLayoutColumn objects"

       type="java.util.ArrayList"%>

      

       <%@ attribute name="jsonTarget" required="true"

    description="json target path"%>

   

    <%@ attribute name="gridDataIdentifier" required="true" description="grid data field identifier, should be the unique key of each row"%>

 

<%@ taglib prefix="c" uri="http://java.sun.com/jsp/jstl/core" %>

 

<script>

dojo.ready(function() {

 

       configure_${id}();

 

});

 

var ${id} = null;

var ${storeId} = null;

 

function configure_${id}() {

 

                /*set the data to the store*/

                ${storeId} = new dojox.data.JsonRestStore({

                      target : "${jsonTarget}",

                      idAttribute : "${gridDataIdentifier}",

                      sortParam : "sortBy",

                      contentType : "application/json",

                      syncMode : true

                  });

 

                 var layout = [ [

                     <c:forEach items="${gridLayout}" var="column" varStatus="status"> 

                                     {

                      name : '${column.columnHeader}',

                      field : '${column.columnField}',

                      width : '${column.width}',

                      <c:if test="${!column.editable==null}">editable : ${column.editable},</c:if>

                <c:if test="${!column.cellType==null}">cellType : ${column.cellType},</c:if>

                <c:if test="${!column.widgetClass==null}">widgetClass :  ${column.widgetClass},</c:if>

                <c:if test="${!column.widgetProps==null}">widgetProps : ${column.widgetProps},</c:if>

                <c:if test="${!column.formatter==null}">formatter : ${column.formatter},</c:if>

                      hidden : ${column.hidden}

                   }<c:if test="${!status.last}">,</c:if>

                   </c:forEach>

                    ] ];

               

    /*create a new grid:*/

    ${id} = new dojox.grid.EnhancedGrid({

       canSort: function(colIndex) {              

            return false;              

        },

        id : '${id}',

        currentRow : null,

        currentCol : null,

        currentDataObject : null,

        store : ${storeId},

        structure : layout,

        rowSelector : '0px',

        plugins : {           

            pagination : {

                pageSizes : [ "25", "50", "100", "All" ],

                description : true,

                sizeSwitch : true,

                pageStepper : true,

                gotoButton : true,

                /*page step to be displayed*/

                maxPageStep : 4,

                /*position of the pagination bar*/

                position : "bottom"

            }

        }

    }, document.createElement('div'));

 

    /*basic row click functionality, make it easier to access currently selected grid object*/

    dojo.connect(${id}, "onRowClick", null, handleSelect_${id});

    function handleSelect_${id}(e) {

         ${id}.currentRow = e.rowIndex;

         ${id}.currentCol = e.cellIndex;

         ${id}.currentDataObject = ${id}.getItem(e.rowIndex);

         //alert( "Row: "+ ${id}.currentRow +" Col: "+ ${id}.currentCol+" Object: "+ ${id}.currentDataObject.${gridDataIdentifier});

    }

   

    /*Call startup() to render the grid*/

    ${id}.store = ${storeId};

    ${id}.startup();

   

    dojo.byId('${id}').appendChild(${id}.domNode);

    ${id}.setStore(${storeId});

 

}

</script>

<br/>

<label><c:out value="${tableHeader}"></c:out></label>

<div id="${id}" style="height: ${height}; width: ${width};"></div>

 

In this case, all the complexity of the ui component is hidden within the tag file. Tag files can make use of custom java objects, as this one does with @ attribute name="gridLayoutColumns" which is an object we created called GridLayoutColumn to simplify defining what we want our table to look like.

Here is the code in the page java Controller that defines our table:

 

final List<GridLayoutColumn> columns = new ArrayList<GridLayoutColumn>();

       columns.add(new GridLayoutColumn("ID", "id", "0px", true));

columns.add(new GridLayoutColumn("Supply Class", "supplyClass", "290px", false));

columns.add(new GridLayoutColumn("Allowance", "allowanceQty", "100px", false));     

…

       …

      model.addAttribute("gridLayout", columns);

 

In this way, we define what a table in our example project should look like. We can define what any component should be and what it should look like. The css is applied inside of the tag so the developer that uses the tag never has to be concerned with what the component should look like or worry about matching a page’s component to look exactly like another page’s component. They’ll all be using the same set of page templates and ui component templates.

 

 Of course, there could be different needs so we can create multiple versions of each ui component.
