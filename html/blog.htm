DOJO vs JQUERY: FIGHT!

Oct 29, 2013

The following is pieced together from research, blogs, and posts around the web. It is an amalgamation of various articles and ideas, and not all of the words or images are mine. My sources are listed at the end.

DOJO vs JQUERY: FIGHT!


“In Dojo I've seen some of the crappies [sic] JavaScript code in my life…” –wise random internet person via StackOverflow

“If you consider yourself technologically impaired, you are probably a better candidate for jQuery” –some jQuery-hatin’ blog

On StackOverflow there are 238,454 questions tagged with jQuery and 4,411 tagged with Dojo. And a large number of those dojo questions go like so:

Question: “How do I get x, y, z to work correctly in dojo?”

Top Answer: “Use jQuery.”

But it is really fair to compare Dojo and jQuery? JQuery is a JS DOM manipulation library, and Dojo is an actual JS framework. Really, jQuery doesn’t do anything dojo can’t . But, which one is more useful and less annoying to code for? Let’s find out!

jQuery Pros

  • Fast
  • Well documented and has a huge community
  • Easy to use
  • Chaining
  • It doesn't extend an object if you didn't specifically ask for it
  • Easy-to-use Ajax
  • Nice event handlers
  • CSS selectors
  • Filtering your selection
  • Did I mention chaining?
  • Small (only 19 KB)
  • Nice little built-in effects.
  • Plugins
  • You don't have to deal with the tangled mess that is cross-platform Javascript compatibility. You don't have to worry about which browsers support standard event handlers and which have their own event systems. You don't have to write two hundred lines of DOM manipulation withHugeLongFunctionNamesFromHell to get nice dynamic pages.
  • You get what you want done in much much much less code, with a lot less hassle
  • One of jQuery’s killer features is its compatibility with many web languages like PHP, ASP, CGI, Servlet etc

VS

Dojo Pros

  • OOP (and other paradigms) done right.
  • Widget infrastructure done right.
  • Modules done right with all necessary goodies:
    • Lazy loading of modules dynamically.
    • Possibility to extract only necessary modules and build a custom one-file profile.
    • Asynchronous loading of modules if desired.
    • Simple integration with CDNs for heavy-duty web applications.
  • Sheer breadth of available modules in DojoX including graphics, charting, grids, and so on.
  • Ability to use it in non-browser environments.
  • Attention to details in widgets:
    • support for i18n (including LTR and RTL languages),
    • support for l10n (including standard date, currency, number formatting),
    • provisions for people with special needs (automatic high-contrast mode, keyboard-only support, and so on) — useful for regular users too, and mandatory for most government contracts.
  • Has all the components you would most likely use (i.e Calendar, Menus, Trees, Validation, Grids, Event Propagations, Layouts etc)
  • Supports different themes (i.e Tundra, Soria etc). Also you can write your own look and feel and customize accordingly
  • Executes fastest on the browser http://blog.creonfx.com/javascript/dojo-vs-jquery-vs-mootools-vs-prototype-performance-comparison
  • Is more mature than its competitors (i.e it has enterprise grade quality)


jQuery Cons

  • You are being abstracted away from the raw Javascript. Don't underestimate the effect of this "con". Though working in vanilla JS (that is, javascript without libraries), can be a massive pain, it gives you a much better understanding of what you're actually doing. You might find that your jQuery based solutions could actually be done in vanilla JS with a lot less overhead.
  • Not quite as fast browser execution as dojo, but still pretty fast.

VS

Dojo Cons

  • And this is the biggest one: Absolutely terrible documentation. More than half of the example code on their website does not work or certain importantly functionality is omitted. This wouldn’t be so bad if for the lack of a vibrant community who would like to share their hacks and code examples.
  • The library is around 1MB of size (a problem if your clients doesn't have good bandwidth)
  • Code easily becomes jumbled, unreadable or hack-ish due to difficultly finding the ‘right’ way to do something in dojo due to the documentation issue. Code flows less elegantly than jQuery.
  • You are writing more code to do what should be simple things.
  • The grid components are lackluster and have to be vastly overwritten and customized to be usable for more complex requirements.


So when it comes to jQuery vs Dojo, we may initially conclude that, jQuery is better than Dojo because:

  • Dojo provides many customizing options but it is a bulky and heavy toolkit whereas jQuery is a light weight with 19K compressed base libraries.
  • Dojo requires a higher network bandwidth than jQuery.
  • At present, the Dojo toolkit is not well documented and the tutorials are less extensive, making it difficult to get in-depth help with development. For jQuery, as many documents and tutorials are easily available out there, it will be the best option to get started.

Ok, so now that we’ve read through all those words, let’s look at some actual code examples and see what’s what:

Hello from Dojo!

<!DOCTYPE html>

require(["dojo/dom", "dojo/domReady!"], function(dom){

</head>

</body>

Hello from jQuery

<!DOCTYPE html>
<html>
<head>
<meta charset="utf-8">

<!-- load JQuery -->

Already we see one big difference in ease of use in the above. jQuery’s $ is ready to use with just the jquery.js included. Dojo has to do a require(["dojo/dom", "dojo/domReady”, blah blah blah for EVERY dojo related object it needs to use.

Alright, so simple things are not really that different. But, how about some AJAX? Say there is a drop down <selection> field, and when selected (and upon window loading) it should prefill the read-only box below it based upon which team is listed.

Dojo AJAX get Manager for Team

<script type="text/javascript">
require(["dojo/parser", "dijit/form/Select", "dijit/form/TextBox"]);
require(['dojo/dom', 'dojo/domReady!', function(dom) {
getManager(dom.byId('team_name').value);
});
function getManager(team) {
require(['dojo/request/xhr', 'dojo/dom'], function(xhr, dom) {
xhr.post('getManager.php', {
method: 'POST',
data: {team: team}
}).then(function(manager) {
dom.byId('manager_name').value = manager;
}, function(error) {
console.error('couldn\'t fetch manger!');
});
});
}</script>
<select name="team" id="team_name" onchange="getManager(this.value)" data-dojo-type="dijit/form/Select">
<option value="TN">Tennessee</option>
<option value="FL">Florida</option>
<option value="CA">California</option>
</select>
<input name="manager_name" id="manager_name" type="text" data-dojo-type="dijit/form/TextBox" readonly/>

jQuery AJAX get Manager for Team

<script type="text/javascript">
$(document).ready(function(){
getManager($("#team_name").val());
});
function getManager(team) {
$.ajax({
type: "POST",
url: "getManager.html",
data: {team:team}
}).done(function( manager ) {
$("#manager_name").val(manager);
}).fail(function() {
console.log( "error" );
}).always(function() {
console.log( "complete" );
});
}
</script>
<select name="team" id="team_name" onchange="getManager(this.value)" type="text" ><option value="TN">Tennessee</option>
<option value="FL">Florida</option>
<option value="CA">California</option>
</select>
<input name="manager_name" id="manager_name" type="text" readonly/>

Notice how there is waaay more code on the Dojo side? And there is more setup involved with AJAX use. jQuery is just $.ajax, dojo you have to define xhr, then have a certain format for xhr.post, dojo.xhrGet, etc.

Now, let’s get to what is often the heart of a web app: the grids. The ability to adequately display and edit your data in a way that fulfills your requirements is so important that if the framework or library you pick is severely lacking enough in that area to where you have to hack up the component to get it to do what you want it to do… you might want to rethink what you are using.

How do the jQuery and Dojo datagrids stack up?

jQuery has a few grid plugins, since jqGrid appears to be the most popular, will look at that one first for the jQuery side.

Dojo has a couple too, but we’ll look at the EnhancedGrid to start with.

Basic Dojo Enhanced Grid using local array list data

<!DOCTYPE html>
<html >

<link rel="stylesheet" href="../../_static/js/dojo/../dijit/themes/claro/claro.css">

/*Grids need an explicit width/height by default*/

width: 45em;
height: 20em;
}
</style>
<script>dojoConfig = {parseOnLoad: true}</script>
<script src='../../_static/js/dojo/dojo.js'></script>
<script>
dojo.require("dojox.grid.EnhancedGrid");
dojo.require("dojo.data.ItemFileWriteStore");
dojo.ready(function(){
/*set up data store*/
var data = {
identifier: 'id',
items: []
};
var data_list = [
{ col1: "normal", col2: false, col3: 'But are not followed by two hexadecimal', col4: 29.91},
{ col1: "important", col2: false, col3: 'Because a % sign always indicates', col4: 9.33},
{ col1: "important", col2: false, col3: 'Signs can be selectively', col4: 19.34}
];
var rows = 60;
for(var i=0, l=data_list.length; i<rows; i++){
data.items.push(dojo.mixin({ id: i+1 }, data_list[i%l]));
}
var store = new dojo.data.ItemFileWriteStore({data: data});
/*set up layout*/
var layout = [[
{'name': 'Column 1', 'field': 'id'},
{'name': 'Column 2', 'field': 'col2'},
{'name': 'Column 3', 'field': 'col3', 'width': '230px'},
{'name': 'Column 4', 'field': 'col4', 'width': '230px'}
]];
/*create a new grid:*/
var grid = new dojox.grid.EnhancedGrid({
id: 'grid',
store: store,
structure: layout,
rowSelector: '20px'},
document.createElement('div'));
/*append the new grid to the div*/
dojo.byId("gridDiv").appendChild(grid.domNode);
/*Call startup() to render the grid*/
grid.startup();
});
</script>
</head>
<body class="claro">
<div id="gridDiv"></div>
</body>
</html>

In conclusion, the dojo EnhancedGrid isn’t terrible, but it IS when you compare it to the jqGrid. jqGrid does what the EnhancedGrid does, but it does it better, and with less unwieldy code. jqGrid is so much simpler to customize and there is MORE to customize. jqGrid wins that fight.

But wait… there is a new grid on the horizon, called dgrid. This new shiny dojo grid that is still in beta looks sharp and has nice clean code! What madness is this? Dojo with clean code?!

Check it http://dojofoundation.org/packages/dgrid/tutorials/hello_dgrid/ , if Dojo 2 (in comparison to current Dojo 1.7, etc) is as smooth looking as this new grid component (in ease of use and extensive features), Dojo has the potential to go from somewhat annoying to something truly awesome to use.

Moving on… to debugging:

Debugging JavaScript can be a tedious and frustrating chore. To compound the already difficult task of debugging, browser vendors each have their own style of error messaging, some of which are confusing, cryptic, or downright misleading to the untrained eye.

Dojo 1.7 was a major pain to debug, especially when the code is optimized to one line. Seeing ‘Error: line 15’ and a blank screen where there should be a couple of grids is not helpful.

Is jQuery as bad?

Not at all, there is a very nice debugging tutorial here for jQuery: http://msdn.microsoft.com/en-us/magazine/ee819093.aspx

And actually, as long as you have Firebug, the newest version of dojo has decent debugging as well: http://dojotoolkit.org/reference-guide/1.9/quickstart/debugging.html

As far as debugging, jQuery and Dojo get a tie.

CONCLUSIONS ?

I really thought going into this comparison that this would really end up being about why Dojo was the worst thing ever and why jQuery(which I had heard so many awesome things about) was awesome. In reality, I found (especially with Dojo’s latest versions) that jQuery and Dojo really aren’t all that different. They are both sound tools to utilize. But, to this first round look at comparing the current state of Dojo and jQuery, I have to give jQuery the advantage. The main tipping point is really Dojo’s poor documentation and lack of a huge community behind it.

I hope in my next round of "jQuery vs Dojo" to take a deeper look at what Dojo 2.0 is going to be like (especially the high potential of the dgrid). If Dojo can pull it off, jQuery just might lose the next round.

ROUND ONE WINNER: JQUERY
